**Goal:** Close the spec's one true backend gap — `GET /admin/users/{id}/trajectories` (wiring the EXISTING `TrajectoryDashboardService::getProgressData` + `TrajectorySelection` read methods, no new compute) — render the Dossier trajectory section into the 1D Dossier shell (whose `trajectories: []` state slot already exists), and add the read-only Trajecten tab (Surface 4) over the already-built `/admin/trajectories(+/{id})` endpoints — while strangling `getTrajectories`/`getTrajectory`/the new wiring out of `AdminAPIController` into a thin `AdminTrajectoryService`.

The Dossier trajectory section and the Trajecten tab are additive Alpine + template partials in the existing `strideApp` / `dashboard.php`.

**Tech Stack:** WordPress (Bedrock) · NTDST Core (DI, repositories) · `stride-core` mu-plugin · Alpine.js · existing `this.api()` `X-WP-Nonce` fetch helper · Playwright (acceptance, `playwright.config.ts` at repo root) · PHPUnit integration suite.

### DRIFT found (act on these)

1. **The phase-map's strangle hazard hypothesis for `getTrajectories` is WRONG — no hazard helper needs relocating.** The phase-map §1 hazard list flags `buildCourseTaxonomyJoin`/`fetchTaxonomyTerms`/`fetchPostTitles` as helpers `getTrajectories` "may use." Ground-truth (read `:1668-2006`): `getTrajectories` uses **NONE** of them. It uses **inline `$wpdb`** (16 `$wpdb` calls — note: MORE than the map's "6", the map undercounted) for the trajectory post list/search, reads trajectory meta (`_ntdst_courses`/`_ntdst_price`/...) inline, and delegates enrollment counts to `RegistrationRepository::findByTrajectoryIds()` (`:97`). `getTrajectory` (`:2012`) is genuinely thin — it just **re-invokes `getTrajectories()` internally with a filter** to fetch one (`:2024`,`:2019`), plus a small `findPendingWithOpenApproval` read. **Routing decision: `AdminTrajectoryService` privately owns the moved `getTrajectories`/`getTrajectory` SQL — NO helper goes to `AdminBatchHelpers`** because none of `getTrajectories`' inline SQL is shared with another domain (verified: its `$wpdb` is trajectory-post-specific; the only cross-domain call is already-extracted `RegistrationRepository::findByTrajectoryIds`). This confirms the instruction's warning that "the 1D S2 implementer found the plan's hazard list was partly wrong — verify, don't trust." **The 1D plan's own hazard prose said S1 needed no hazard relocation either; this is the same pattern.**

2. **`getTrajectories` carries inline trajectory-meta parsing that is INV-3-adjacent but pre-existing — move VERBATIM, do not "fix" during the strangle.** It reads `_ntdst_courses` (hardcoded meta key, `:62`/`:73`) and json-decodes it inline. This is a pre-existing drift (hardcoded `_ntdst_*` prefix, INV-3) but the strangle is **behavior-preserving** — move it as-is into `AdminTrajectoryService`. A drift-cleanup of trajectory meta reads through `TrajectoryRepository` is **out of scope** for 1E (it would change behavior and risk the green-before/after net). Note it for a future trajectory-domain pass; do NOT bundle it here.

3. **The Trajecten TAB is PARTLY built already (1D side-effect) — 3.6 is smaller than the spec implies.** The spec frames Surface 4 as "a frontend tab over existing endpoints" net-new. Ground-truth: 1D's `admin-dashboard.js` ALREADY ships `loadTrajectories()`/the detail fetch/`trajectoryCourses`/`trajectoryRegistrations` getters/`trajectoryFilters`/`trajectoryPagination` (js:170-264, 781-817). Confirm at execution which getters exist and build only the gap.

4. **`getUserTrajectories` needs a user→trajectory-ID discovery step the spec did not name — but it ALREADY EXISTS, no new repo method.** The spec says "for each trajectory the user is in, return progress." It does not name HOW you get "the trajectories the user is in." Ground-truth: `EnrollmentService::getEnrolledTrajectoryIds(int $userId): array` (`:826`) returns exactly that (the user's enrolled trajectory IDs, the same set `hasActiveRegistration`/`existsForTrajectory` use), and `RegistrationRepository::findTrajectoryEnrollmentsByUser(int $userId)` (`:1152`) returns the parent rows. **So `getUserTrajectories` = `getEnrolledTrajectoryIds($userId)` → loop `getProgressData($userId,$trajId)` + selection reads. Zero new SQL, zero new repo method** — pure wiring, exactly as the spec promised. `tab-trajecten.php` is the live precedent for this discovery (`findTrajectoryEnrollmentsByUser` → per-trajectory render).

5. **The endpoint must also return the trajectory's `status`/`mode`/`title` (the Dossier section header needs them) — `getProgressData` returns `mode` but NOT `title`/`status`.** `getProgressData` returns `mode` (`:122`) but the Dossier mockup header (`dossier.html:102-105`) binds `t.trajectory.title`, `t.trajectory.status`, `t.trajectory.mode`. So `getUserTrajectories` composes a `trajectory:{id,title,status,mode}` sub-object from `TrajectoryService::getTrajectory($trajId)` (the live `tab-trajecten.php` does exactly this, `:43`) PLUS the `getProgressData` payload. Name this in the response contract (Task 3.5) so the implementer composes both, not just `getProgressData`.

6. **The `elective_groups` shape from `getProgressData` is the RAW group definition, not the chosen-vs-required view the mockup binds.** The mockup `dossier.html` (`:143-170`) binds per group: `g.name`, `g.required`, `g.total`, `g.countChosen`, `g.isChosen`, `g.chosen[]` (`{title,edition}`). `getProgressData`'s `elective_groups` is the raw group definition (name/required/courses). The endpoint must **enrich** each group with `countChosen`/`isChosen`/`chosen[]` by running `getSelectedCourseIds($parentRegId)` once then `countChosenInGroup`/`isGroupChosen` per group (exactly `tab-voortgang.php:65,86-98`). This enrichment IS the "chosen-vs-required" the Tier-A contract asserts — it is the real (small) composition work, not a pass-through. The parent registration id for `getSelectedCourseIds` comes from `findTrajectoryEnrollmentsByUser` (the parent row's `id`).

**Net:** no premise is fatally wrong. The backend gap (`getUserTrajectories`) is real and is genuinely wiring — but the wiring has two named composition steps the spec glossed (DRIFT #5 title/status/mode header, #6 elective enrichment), both mirrored verbatim from `tab-voortgang.php`/`tab-trajecten.php`.
