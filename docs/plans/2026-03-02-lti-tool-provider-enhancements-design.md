# LTI Tool Provider Enhancements — Design

**Date:** 2026-03-02
**Scope:** Tool Provider side only. Platform side deferred.
**Target platforms:** Moodle, Canvas, Blackboard, and others (broad LTI 1.3 compatibility).

---

## 1. Config Endpoint

Serve tool configuration as JSON and XML so LMS admins can auto-configure the tool by pasting a URL.

### Routes

- `GET /lti/configure.json` — IMS LTI 1.3 JSON config (Moodle, most LMSes)
- `GET /lti/configure.xml` — Canvas-specific XML config format

### JSON Schema (IMS Tool Configuration)

```json
{
  "title": "Stride LMS",
  "description": "LearnDash course delivery via LTI 1.3",
  "oidc_initiation_url": "https://stride.ddev.site/lti/login",
  "target_link_uri": "https://stride.ddev.site/lti/launch",
  "jwks_uri": "https://stride.ddev.site/lti/jwks",
  "claims": ["sub", "name", "email", "given_name", "family_name"],
  "messages": [
    { "type": "LtiResourceLinkRequest" },
    { "type": "LtiDeepLinkingRequest", "target_link_uri": ".../lti/deep-link" }
  ],
  "scopes": [
    "https://purl.imsglobal.org/spec/lti-ags/scope/lineitem",
    "https://purl.imsglobal.org/spec/lti-ags/scope/score"
  ]
}
```

Canvas XML wraps the same data in Canvas's proprietary format.

### Implementation

Add two new methods to `ToolProvider\Router` (~60 lines each). No new classes.
New routes: `configure-json` and `configure-xml` actions on the existing rewrite rule.

## 2. Dynamic Registration

Automated tool registration with LMS platforms. No manual copy-pasting of keys and endpoints.

### Flow

```
1. Platform sends GET /lti/register?openid_configuration=...&registration_token=...
2. Tool fetches platform's OpenID config from that URL
3. Tool displays confirmation page to admin (platform name, endpoints)
4. Admin clicks "Register"
5. Tool POSTs its config to platform's registration endpoint
6. Platform responds with client_id + deployment_id
7. Tool creates lti_platform CPT entry with all details
8. Platform is configured, ready for launches
```

### Implementation

- Leverage `ceLTIc\LTI\Tool::doRegistration()` for protocol handling
- New route `/lti/register` in `Router.php` — handles GET (initiation) and POST (confirmation)
- Override `onRegistration()` in `Tool.php` — renders confirmation page
- New template `templates/registration-confirm.php`
- `WPDataConnector::savePlatform()` already handles persistence

### Security

Only logged-in admins with `manage_options` can confirm registration. Anonymous visitors see an error.

### Action Hook

`netdust_lti_platform_registered` fires after successful registration with the platform post ID.

## 3. Per-Platform Role Mapping

Each platform can define which WP roles LTI Instructor and LTI Learner map to.

### Data Model

Two new fields on `lti_platform` CPT:

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `role_instructor` | `select` | `instructor` | WP role for LTI Instructor |
| `role_learner` | `select` | `subscriber` | WP role for LTI Learner |

Options populated from `wp_roles()->get_names()`.

### Changes

- **`LTIDataService`** — Add fields + "Role Mapping" field group to `lti_platform` registration
- **`UserProvisioner::provision()`** — Accept platform post ID, read role mapping from meta, fall back to defaults
- **`Tool::onLaunch()`** — Pass platform ID to `UserProvisioner`

No new classes. Metabox UI auto-generated by Data Manager.

## 4. GradePayload Value Object

Replace scattered scalar parameters in `GradePassbackService` with a typed, immutable object.

### Class: `ToolProvider\Domain\GradePayload`

```php
readonly class GradePayload
{
    public function __construct(
        public int $userId,
        public int $courseId,
        public float $score,
        public float $maxScore,
        public string $activityProgress,  // 'Completed', 'InProgress'
        public string $gradingProgress,   // 'FullyGraded', 'Pending'
        public ?string $comment = null,
    ) {}

    // Named constructors
    public static function completion(int $userId, int $courseId): self;
    public static function quizScore(int $userId, int $courseId, float $score, float $max): self;
    public static function tincannyScore(int $userId, int $courseId, float $percentage): self;
}
```

### Changes

- **`GradePassbackService`** — Three public methods collapse to one: `postGrade(GradePayload $payload): bool|WP_Error`
- **`LearnDashBridge`** / **`TinCannyBridge`** — Use static factories on GradePayload

## 5. Username Scopes (Simplified)

Email-based identity: same email across platforms = same WP user. No full scope system needed.

### Changes to UserProvisioner

1. **Deterministic username generation** — Standardize on `{given}.{family}` (sanitized), falling back to email prefix, then `lti_{sub_hash}`. Consistent and predictable.

2. **Platform-aware sub matching** — Store `_netdust_lti_sub` as `{platform_id}:{sub}` instead of bare `sub`. Two platforms could send the same sub value for different users. Email fallback still provides cross-platform identity.

3. **Filter hook** — `netdust_lti_provision_user_data` on the user data array before `wp_insert_user()`.

### Migration

Existing `_netdust_lti_sub` values (bare subs) continue to match — the lookup checks both `{platform_id}:{sub}` and bare `{sub}` formats. New entries always use the scoped format. No migration script needed.

## 6. Filter Hooks (Minimal)

Six hooks at key decision points. No speculative hooks.

### Launch Flow (3 filters)

| Hook | Location | Signature | Purpose |
|------|----------|-----------|---------|
| `netdust_lti_claims` | `Tool::onLaunch()` | `(LtiClaims $claims): LtiClaims` | Modify parsed claims before processing |
| `netdust_lti_provision_user_data` | `UserProvisioner::createUser()` | `(array $data, LtiClaims $claims): array` | Modify user data before wp_insert_user |
| `netdust_lti_should_enroll` | `CourseEnroller::enroll()` | `(bool $should, WP_User $user, int $courseId, LtiClaims $claims): bool` | Skip enrollment (capacity, prereqs) |

### Grade Flow (2 filters)

| Hook | Location | Signature | Purpose |
|------|----------|-----------|---------|
| `netdust_lti_grade_payload` | `GradePassbackService::postGrade()` | `(GradePayload $payload): GradePayload` | Modify grade before submission |
| `netdust_lti_should_post_grade` | `GradePassbackService::postGrade()` | `(bool $should, GradePayload $payload): bool` | Suppress grade passback |

### Registration (1 action)

| Hook | Location | Signature | Purpose |
|------|----------|-----------|---------|
| `netdust_lti_platform_registered` | Dynamic registration | `(int $platformPostId, Platform $platform): void` | Post-registration notification |
