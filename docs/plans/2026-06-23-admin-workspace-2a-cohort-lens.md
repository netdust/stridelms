# Admin Workspace — Phase 2a: Cohort Lens Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the **Editie/sessie cohort lens** of the Stride Admin Workspace — the second of three deferred slices. From an edition, drill into its sessions → a **per-session roster** ("who is in which session"), mark **attendance per session**, **filter/count roster extras** (book/lunch/dietary, loaded-set only), run **roster bulk actions** (inheriting the Phase-1 `RegistrationTransitions` map), run **trajectory-roster bulk actions** (deferred from Phase 1), and surface the **5 existing Edition exporters** on the roster. The §13 edition-lifecycle task engine is **explicitly OUT of scope** (Phase 2b).

**Architecture:** A new per-edition roster read-model (`GET /admin/editions/{id}/roster`) assembles the edition's loaded registration set + each registrant's **session selections** (read through the `SessionSelection` / `RegistrationRepository::getSelections()` convergence point — never the raw `selections` column) + **attendance** (via `AttendanceRepository::getByUsers`) + **extras** (from `enrollment_data` JSON, **loaded-set only**, never a global grid column). Attendance writes go through the existing `POST /admin/attendance` route + `AttendanceService`. Roster bulk + trajectory-roster bulk register on the existing `ntdst/v1/action` registry, reusing the Phase-1 `BulkRegistrationHandler` per-row-report + capability-before-loop pattern. The editions/sessions read SQL strangles out of `AdminAPIController` into a new `AdminEditionRosterService`.

**Tech Stack:** WordPress (Bedrock) · NTDST Core (`ntdst/api_data` registry, DI, repositories) · `stride-core` mu-plugin · Alpine.js + the existing `admin-dashboard.js` · `wp_vad_registrations` + `wp_vad_attendance` custom tables · `vad_session` CPT · the 5 `Modules/Edition/Admin/Edition*Exporter` exporters · PHPUnit (Unit + Integration) + Playwright/Chrome acceptance.

**Work classification:** **Class A** — new multi-surface feature (per-session roster + attendance writes + two bulk surfaces + exporter wiring + editions/sessions god-class strangle), full Stage 0→1 gated plan. Stage 0 brainstorming was pre-resolved by the spec (§1 cohort lens, §534 Phase 2 row, §11/§324 trajectory-roster). This document is the Stage-1 gated plan, ground-truthed against current source (HEAD `5e1befb7`).

## Threat model

> For the Phase 2a cohort lens, written 2026-06-23 at plan-time (proactive, before task breakdown). Cohort inherits spec M1–M7 for roster bulk (§489 "the cohort lens inherits M1–M7 for its roster bulk actions") and M8 for the exporter boundary — **those are cross-referenced, NOT re-litigated**. This section ADDS the cohort-specific attack classes the global grid did not have: a roster scoped to ONE edition introduces cross-edition leakage, attendance for a session outside the edition, and extras-JSON leaking onto a global surface. This section is the convergence target — `/code-review` + `security-sentinel` verify against the numbered mitigations.

### What we're defending

- **CA1 — Per-edition roster confidentiality.** One edition's full participant roster (names, organisations, attendance %, session selections, extras like dietary needs) assembled in one payload. Extras (dietary/medical-adjacent) are sensitive-category PII. The roster is a per-edition aggregation surface.
- **CA2 — Attendance-record integrity.** `vad_attendance` writes carry side effects (audit events, auto-completion via `AttendanceService`). A mis-scoped or forged attendance write corrupts a registrant's completion state.
- **CA3 — The `enrollment_data` extras stage** (dietary, lunch, book) — must stay **per-edition loaded-set** and must NOT bleed into the global Inschrijvingen grid (the M4/M5 structured-only contract) or out to an external party except through the deliberate, anonymise-skipped exporter (M8).
- **CA4 — Cross-edition / cross-trajectory isolation.** A roster bulk action or attendance write must affect ONLY the edition (or trajectory child-set) the actor is acting on — not a registration id smuggled from another edition/trajectory.
- **CA5 — Server resources / the registration+attendance tables.** A roster endpoint with session/extras params is a new path to expensive queries / injection if params reach SQL un-whitelisted.

### Who we're defending against

- **External unauthenticated attacker** — IN scope (must hit `permission_callback` and get nothing).
- **`stride_supervisor` (view-only) user** firing a roster bulk action or an attendance write — IN scope (confused-deputy; attendance is `canManageAdmin`, bulk is `canManageAdmin`).
- **A `stride_coordinator` (manage) user CSRF'd** into a roster bulk mutation — IN scope (registry Origin/Referer + nonce).
- **A `stride_coordinator` who legitimately manages edition A** crafting a payload with **edition-B / trajectory-B registration ids or a session id outside edition A** — IN scope (the cohort-specific per-row scope; the global cap is `stride_manage` but the roster surface implies an edition scope the payload must not escape).
- **A partner user** (`partner` role) reaching the admin roster — IN scope (denied; admin roster is not partner-facing).
- **Insider with valid `stride_coordinator` credentials acting within their grant** — OUT of scope (audit-trail only; `stride_manage` is global across editions by design — see deferrals).
- **DNS rebinding / network-position** — n/a (no user-controlled outbound URL in this feature).

### Attacks to defend against

> Inherited (cross-ref, do NOT re-discover): **M1** unauth read · **M2** view-only mutation · **M3** per-row authz skipped · **M4** SQL-injection/expensive-query via params · **M5** JSON column in WHERE/GROUP BY · **M6** CSRF on bulk · **M7** `set_field` lifecycle bypass (n/a here — no roster set-field) · **M8** invoice/PII egress via export. The spec's `## Threat model` (`docs/plans/2026-06-13-admin-workspace-spec.md`) is the source for M1–M8.

Cohort-specific NEW attacks:

1. **C-ATK-1 — Cross-edition / cross-trajectory roster-bulk leak.** A `stride_manage` actor opens edition A's roster, then POSTs a `stride_roster_bulk_*` payload containing registration ids that belong to edition B (or a trajectory the actor wasn't acting on). Because the global cap is `stride_manage`, a *naïve* per-row check (existence + `stride_manage` only) would mutate the foreign rows — the roster's implied edition scope is escaped. (This is the cohort manifestation of the traverse-clause/confused-deputy class: the "what may this roster action touch" decision is re-derived per call site instead of bound to the opened edition.)
2. **C-ATK-2 — Attendance for a session not in this edition.** A `canManageAdmin` actor (or a forged request that passes the cap) marks attendance with a `session_id` that belongs to a DIFFERENT edition than the `user_id`'s registration — corrupting an unrelated edition's attendance / triggering a wrong auto-completion. Today `markAttendance` validates the session *exists* and the user *exists*, but **does NOT verify the session belongs to the same edition as a registration the user holds**.
3. **C-ATK-3 — Extras-JSON leak onto a global surface.** A well-meaning change surfaces an extras field (dietary/book) as a global Inschrijvingen grid column or filter — binding `enrollment_data` JSON into a WHERE/GROUP BY at corpus scale (M5 violation) AND exposing per-edition sensitive data on a surface that isn't edition-scoped.
4. **C-ATK-4 — Exporter as an authz/PII bypass.** The roster exposes a "download" affordance that calls an exporter with an attacker-chosen `type` (class name) or an edition id outside the actor's intent, or that skips the `_stride_anonymised_at` guard — egressing the full (non-field-scoped) roster incl. PII.
5. **C-ATK-5 — Roster read expensive-query / injection via session/extras params.** The roster endpoint accepts `session_id` / `extras_key` / a roster filter that, if interpolated or bound to JSON, injects or table-scans.
6. **C-ATK-6 — Double-mark / concurrent attendance race.** Two coordinators mark the same `(session, user)` concurrently, or a bulk roster confirm overlaps a single confirm — double events / inconsistent state. (Attendance `record()` should be upsert-idempotent; bulk reuses the idempotent single-item path.)

### Mitigations required (numbered to match attacks; M1–M8 inherited)

1. **CM-1 — Edition/trajectory-scoped per-row authorization on roster bulk.** Every `stride_roster_bulk_*` handler takes the **opened scope** (`edition_id`, or `trajectory_id` for the trajectory-roster) as a REQUIRED param, and each row id in the loop is verified to belong to that scope BEFORE mutation: for the edition roster, `RegistrationRepository::find($id)->edition_id === $editionId` (verified: `find()` is `SELECT *`, returns `edition_id`, `:401`). **⚠️ CORRECTED (B2, security review 2026-06-23): for the trajectory roster, the scope is NOT `findEditionsByTrajectory()` — that method is per-USER (`findEditionsByTrajectory(int $userId, int $trajectoryId)`, `:717`, `WHERE child.user_id = %d`) and cannot serve a multi-user bulk scope.** The correct trajectory-scoped (NOT user-scoped) parent→child join exists only INLINE inside `queryForGrid` (`:2009–2034`, Phase-1B Task 1.4b — "the grid is trajectory-scoped, NOT user-scoped, so the parent.user_id constraint is dropped"). **2a must EXTRACT that join shape into a reusable repository method** (e.g. `findChildRegistrationIdsByTrajectory(int $trajectoryId): array` or `existsInTrajectoryChildSet(int $regId, int $trajectoryId): bool`) — see Task 2a.8 — and the trajectory-roster bulk checks each row against THAT set. A row outside the scope goes to `failed[]` with `out_of_scope`, never mutated. This is layered ON TOP of M2 (`stride_manage` first line) + M3 (existence). **The scope set is computed once from the opened edition/trajectory, not re-derived per row from client input.** Pinned by a RED denial test (a foreign-edition / foreign-trajectory id → `out_of_scope`, not mutated).
2. **CM-2 — Session-belongs-to-edition check on attendance.** `markAttendance` (and the roster attendance UI's calls) verify the `session_id`'s parent edition matches an edition the `user_id` is **registered in**, via `SessionService::getSession($id)` → `edition_id` (`SessionService.php:48`) + **`RegistrationRepository::existsForEdition($userId, $editionId)`** (`:586` — the exact membership primitive; use this, NOT `findByEdition`, per I1), returning `WP_Error('session_edition_mismatch', …, ['status'=>400])` otherwise. **⚠️ Risk restatement (I1, security review): the real residual risk is NOT "corrupts edition B's records" — `AttendanceService::markPresent` resolves the editionId from the session's OWN `edition_id` (`AttendanceService.php:83`), so it always writes to the session's true edition. The actual harm is recording attendance + firing auto-completion side effects for a user against a session in an edition the user is NOT registered in.** The RED test therefore asserts: a `(session, user)` where the user is not registered in the session's edition → rejected (not a write to a wrong edition). Closes the today-gap.
3. **CM-3 — Extras are loaded-set-only, enforced by construction.** The roster read-model reads extras from `enrollment_data` JSON **only for the page of rows already loaded for this edition** (the `summarizeEnrollmentData` pattern, D3), and exposes NO param that binds an extras key into a SQL WHERE/GROUP BY. The global `RegistrationRepository::queryForGrid()` (Phase 1) is NOT touched to add an extras param. The §675 sibling-audit pins this: a reviewer seeing an extras key reach `queryForGrid`/`queryForGridGrouped` flags it. (Inherits M5.)
   - **CM-3b — Roster READ anonymise decision (I2, security review 2026-06-23 — make it explicit, not silent).** The roster READ payload (`GET /editions/{id}/roster`) includes names/org/attendance/extras (extras are sensitive-category PII, CA1) and is reachable by a `stride_view` supervisor. **Decision for 2a: the read roster DISPLAYS `_stride_anonymised_at` users with their PII fields redacted/tombstoned** (name → "(verwijderd)", extras suppressed), consistent with the export boundary — an erased user must not reappear in full on an admin read surface. This is a small per-row check in `AdminEditionRosterService` reusing the same `_stride_anonymised_at` flag. Pinned by a Task 2a.1 assertion (an anonymised registrant in the edition → roster row present but PII-redacted). NOT a deferral — it is built in 2a alongside the export skip, from the same flag.
4. **CM-4 — Exporter wiring uses a server-side type allowlist + capability + a UNIVERSAL anonymise-skip (NOT "as-is").** The download route maps a fixed `{type}` ∈ `{attendance, registration, namecard, bundle, files}` to the corresponding exporter class server-side (never a class name from the request), gates on `canManageAdmin` (PII egress), validates `{id}` via `absint` + edition-exists. **⚠️ CORRECTED (B1, security review 2026-06-23 — verified against source): the `_stride_anonymised_at` skip exists on ONLY `EditionFilesZipExporter` (`:76`). The other 4 (`EditionRegistrationExporter`, `EditionAttendanceExporter`, `EditionNamecardExporter`, `EditionBundleZipExporter`) have ZERO anonymise skip.** Surfacing all 5 "as-is" as one-click roster downloads behind only `canManageAdmin` would egress the PII of GDPR-erased users through 4 exporters that ignore the flag — the spec puts *universalising* the skip in Phase 3, but Phase 2a is what newly makes these one-click-reachable from the roster, so 2a OWNS the regression it introduces. **Mitigation for 2a: lift the `_stride_anonymised_at` skip into a shared point all 5 exporters honour (a shared trait/base filter on the participant set), with a RED test per exporter asserting an anonymised user is excluded.** This is a small, contained addition (the skip already exists to copy from `EditionFilesZipExporter:76`) and is the prerequisite that makes the CF6 "boundary" anonymise edge actually pass. Do NOT surface the 4 skip-less exporters until the skip is universal. (Inherits/extends M8; the *field-scoped* allowlist remains Phase 3.)
5. **CM-5 — Server-side whitelist for every roster param.** `session_id`/`edition_id` via `absint`; any `extras_key` filter validated against the keys present in the loaded set (or applied client-side over the returned page); no param string-concatenated into SQL; the roster query method uses `$wpdb->prepare`. (Inherits M4.)
6. **CM-6 — Idempotent attendance + idempotent bulk.** `AttendanceRepository::record()` is upsert-by-`(session,user)` (verify at task start — `findBySessionAndUser` + update-or-insert); roster bulk reuses the Phase-1 idempotent single-item path (`confirmRegistration` guards confirmed→confirmed).
