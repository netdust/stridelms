# Stride LMS — Technical Audit & Improvement Plan

**Date:** 2026-06-10 · **Branch:** staging · **Auditor:** Claude Code (4 parallel specialist audits — security, performance, architecture drift, quality/testing/docs — plus direct verification of the highest-stakes claims)
**Performance numbers measured against the live DDEV database** (28 editions, 128 registrations, 27k audit rows).

> Companion to `docs/LAUNCH-CHECKLIST.md`. Task references (0.x / 1.x / 2.x / 3.x) are this document's milestones, defined in the Task Plan section.

---

## Executive Summary

**Overall health grade: B−.** The codebase is disciplined in ways most WordPress projects never are — architecture invariants enforced in CI, uniform SQL parameterization, zero empty catch blocks, near-universal `strict_types`, genuinely good tests where they exist, and the previously-flagged Critical security issue (impersonation audit bypass) is verified **fixed**. What pulls the grade down is not sloppiness but three structural gaps: ~1,800 tests exist and **CI runs none of them**; the public catalog and logged-in dashboard have **measured** query counts (357–386 per request on dev data) that will not survive 4,000 users without an object cache and batched hydration; and two **verified-broken features** (the registrations CSV export always produces an empty file; an admin action-queue rule silently returns nothing) ship on the launch branch.

**Top 3 risks:**
1. Performance collapse of the dashboard and crawlable catalog at production scale.
2. Regression blindness because the entire test estate runs only on developer machines.
3. The documented deploy path (`make deploy-staging`) references a Makefile that does not exist, days from launch.

**Top 3 opportunities:**
1. Wiring the existing suites into CI is nearly free and converts ~1,800 tests into a real safety net.
2. Batch-hydrating the catalog with the already-built `BatchQueryHelper` cuts the hottest public pages by an order of magnitude.
3. Consolidating five duplicated business constants (VAT rate, status labels, active-status set) plus PHPStan level 1 eliminates the drift class that keeps producing bugs here.

## Repo Map

**Purpose.** Stride is a production-bound LMS for VAD training management (4,000+ users at launch), a clean rewrite of VAD Vormingen v3. Maturity: late pre-launch — ship-mode, hardening sprint phases 0–2 complete.

**Stack.** Bedrock WordPress (PHP 8.3, `roots/wordpress` 6.9.1), LearnDash as content engine, NTDST Core framework (DI container, router, data layer), FluentCRM/Forms/SMTP, DOMPDF, Tailwind + Alpine.js + Vite theme. MariaDB 10.11 via DDEV; Ploi hosting.

**Architecture.** Strict mu-plugin/theme split: `stride-core` (business logic, 148 PHP files, ~43k lines) + `stridence` theme (presentation, 96 files, ~13k lines). Domain modules (Edition, Enrollment, Invoicing, Trajectory, Attendance, …) own repositories; two high-volume custom tables (`wp_vad_registrations`, `wp_vad_attendance`) are owned 1:1 by repositories. Three REST surfaces: Admin (`/stride/v1/admin/*`), Partner (`/stride/v1/partner/*`), Assistant abilities. Frontend AJAX converges on the framework's nonce-verified `ntdst/api_data/*` dispatch.

| Area | One-liner |
|---|---|
| `web/app/mu-plugins/stride-core/` | All business logic: Modules/, Admin/, Handlers/, Domain/ value objects, Infrastructure/ base classes |
| `web/app/themes/stridence/` | Presentation only (verified: zero writes/business mutations in templates) |
| `tests/` | 69 Unit + 42 Integration + 18 acceptance Cests + 20 Playwright specs (~1,800 test methods) |
| `ARCHITECTURE-INVARIANTS.md` + `scripts/check-invariants.sh` | Named convergence points, grep-enforced in CI (INV-1 blocking) |
| `docs/` | Real, current docs: LAUNCH-CHECKLIST.md, FEATURE-STATUS.md, BASELINE-2026-06-08.md, plans/ |
| `bck/`, root `*.zip` | ~90 MB of tracked debris (old Kadence theme backup, SCORM/xAPI packages, screenshots) |
| `.github/workflows/` | ci.yml (invariants + composer validate + Pint), integration.yml (installs WP, curls homepage — no tests) |

**Surprises.** (1) The test estate is large and high-quality yet entirely absent from CI. (2) `composer.json` autoloads `stride\` from `web/app/themes/stride/` — a directory that does not exist. (3) `site.yml` declares a Makefile-based deploy; there is no Makefile. (4) Several project-memory entries are stale in the *good* direction: the impersonation audit bypass and the acceptance-suite prefix mismatch are both fixed.

## Audit Report

Findings merged across the four specialist passes, deduplicated, sorted by severity. **[F]** = verified fact (file read / measured), **[J]** = judgment.

### Critical

**CR-1 — CI runs zero of ~1,800 tests.** [F] `.github/workflows/ci.yml` runs only `scripts/check-invariants.sh`, `composer validate`, and Pint; `integration.yml` installs WP in a devcontainer and asserts `curl localhost:8080 | grep "<title>Bedrock"`. No workflow contains `phpunit`, `codecept`, or `playwright`. Inventory that never runs: 69 unit files (~816+ methods), 42 integration files (~370), 18 acceptance Cests (118 scenarios), 20 Playwright specs (265 tests). *Why it matters:* this repo has already lived the failure mode — the acceptance suite was committed structurally broken and its "126 tests" cited as evidence without ever running (memory: `lesson_suite_counts_need_a_run`). Local-only suites rot silently.

**CR-2 — Dashboard non-home tabs run the full enrollment aggregation twice; measured 357 queries / 399 ms for `getNavData()` alone.** [F] `page-mijn-account.php:35-41` calls `UserDashboardService::getNavData()` (full `getEnrollmentData()` + `getQuoteData()` + trajectory build) just to compute five nav booleans; the tab template then re-runs the same aggregation (`tab-inschrijvingen.php:25`, `tab-offertes.php:24`). `getEnrollmentData()` has no per-request memoization (`UserDashboardService.php:109-116, 274-293`; only `RegistrationRepository::findByUser` is memoized, `RegistrationRepository.php:22, 958-991`). *At scale:* the most-visited logged-in page costs 800–1,500 queries and 1–2 s for an active learner.

**CR-3 — Public catalog pages are N+1 per card and render up to 200 cards server-side; measured 347–386 queries with only 28 editions.** [F] Each card partial queries for itself: `partials/card-edition.php:49-54` (`getEffectiveStatus()` → per-card session-count join, `SessionRepository.php:101-106`), `:60` (`get_post` per card), `:82` (`isEnrolled()` → uncached query per card), `partials/card-course.php:34-47` (a full `WP_Query` with meta_query *per card*). `page-klassikaal.php:37` / `page-online.php:63` render up to 200 cards; Alpine pagination only hides them client-side (`page-klassikaal.php:141-152`). *At scale:* ~1,500–2,500 queries per anonymous, bot-crawlable request, with no page cache in the stack. The `BatchQueryHelper` built precisely for this is used consistently in Admin/ and not at all in the theme.

### High

**H-1 — Two queries reference non-existent columns; both features are silently dead.** [F — verified directly against schema] `AdminAPIController.php:2236-2237` filters `r.tasks` (column is `completion_tasks`, `RegistrationTable.php:49`) — the "incomplete tasks" action-queue rule always returns nothing (`$incompleteTasks ?: []` swallows the SQL error). `AdminAPIController.php:3433` orders by `r.created_at` (column is `registered_at`, `RegistrationTable.php:45`) — **the registrations CSV export always produces an empty file.** *Why it matters:* broken admin features on the launch branch, invisible because errors are swallowed; also proof of what PHPStan/integration tests would catch.

**H-2 — Documented deploy path does not exist.** [F] `site.yml:22-25` declares `deploy.method: makefile` with `make deploy-staging` / `make deploy-production`; no Makefile exists in the repo. *Why it matters:* the project is in deploy-readiness phase; the single documented deployment procedure is currently un-runnable as written.

**H-3 — No persistent object cache; all caching is wp_options transients.** [F] No `object-cache.php` drop-in; `wp_cache_*` unused in stride-core; dev DB already shows 196 stale `stride_edition_reg_count_*` option rows for 28 editions (`EditionService.php:104-123`). *Why it matters:* multiplies CR-2/CR-3 under concurrency (transient stampedes, options-table churn).

**H-4 — Notification badge scans the audit log with un-indexable JSON predicates on every dashboard view.** [F — measured] `page-mijn-account.php:72-73` → `AuditRepository::findBySubjectUser()` (`ntdst-audit/src/AuditRepository.php:182-198`) applies `JSON_EXTRACT(context,…)` over a 30-day range; EXPLAIN shows a 13,640-row range scan (40.7 ms at 27k dev rows). `AuditBridge::onMailSent` (`AuditBridge.php:525-542`) logs every outbound email into the same table. *At scale:* 100–500 ms added to every logged-in page.

**H-5 — Belgian VAT (21%) is implemented in three places, including a full re-implementation of quote totals in the admin discount path.** [F] Canonical: `QuoteCalculator.php:16` (`TAX_RATE = 21.0`). Bypasses: literal `0.21` at `QuoteService.php:236, 654` and `QuoteAdminController.php:504, 549, 568, 614`, where `applyManualDiscount`/`removeDiscount` re-derive subtotal→discount→tax→total independently. *Why it matters:* drift on financial math; the two paths can already disagree.

**H-6 — stride-core calls a theme helper with no guard — fatal on theme swap.** [F] `stride_format_date()` (defined in `stridence/helpers/formatting.php:17`) is called from `NotificationMapper.php:139` and `StrideMailBridge.php:223, 759, 760`, violating INV-5. It slipped the invariant audit because INV-5's grep only matches the `stridence_` prefix. *Why it matters:* client-theme swaps are a real scenario (Kindred); a swap fatals the mail/notification render path.

**H-7 — AdminAPIController: 3,510-line god class with ~1% test coverage of its surface.** [F] 25 REST routes, 156 `$wpdb` references, 65 hardcoded `_ntdst_` literals, ≥12 responsibilities; longest methods `registerRoutes()` (349 lines), `getUserDetail()` (337 lines). Its only direct tests cover `canViewAdmin`/`canManageAdmin`/`sanitizeCsvCell` (`tests/Unit/AdminAPIControllerTest.php`, 144 lines). [J] The decomposition is tracked, post-launch, previously declined — re-litigating it is double-counting. The *new* information is the test gap on the highest-privilege surface in the product, and that both H-1 bugs live here.

**H-8 — `getPendingApprovals` pulls unbounded `SELECT *` with leading-wildcard LIKE over JSON blobs.** [F] `AdminAPIController.php:1970-1984` — no LIMIT, `completion_tasks LIKE '%post_approval%'`, full longtext rows decoded per poll of the admin dashboard. *At scale:* thousands of JSON blobs per poll.

### Medium

**M-1 — Sensitive-PII reveal endpoint is not audit-logged.** [F] `AdminAPIController.php:2930-2950` returns `national_id`, `date_of_birth`, license number in cleartext with no audit row (a comment at ~2925 says "audit-friendly later" — deferred and never added), while impersonation *is* audited. GDPR special-category access with zero forensic trail. One-line fix at an existing convergence point.

**M-2 — Completion-proof uploads land in the public media library.** [F] `CompletionTaskHandler.php:169` stores ownership- and MIME-validated uploads as standard attachments — unauthenticated, guessable `/uploads/YYYY/MM/` URLs for documents that may contain certificates/IDs.

**M-3 — Partner API has no rate limiting on user-creation/enrollment.** [F] `PartnerAPIController.php:614-656`; blast radius bounded by company scoping, but a leaked partner credential can mass-create real WP users and trigger mail/LMS side effects.

**M-4 — `enrollment_path` ENUM is missing `'partner'`.** [F] `RegistrationTable.php:39` defines `ENUM('individual','colleague','trajectory')` but `RegistrationRepository::PATH_PARTNER = 'partner'` (`:20`). Partner enrollments insert as empty string (non-strict) or fail (strict).

**M-5 — Parallel registrations-query pattern in `Modules/Edition/Admin/`: the same "registrations for edition + usermeta" SQL re-implemented 5 times outside RegistrationRepository, keyed off a duplicate table-name constant.** [F] `EditionAdminController.php:32, 1229`; `EditionRegistrationMetabox.php:590, 617`; `EditionNamecardExporter.php:168, 191`; `EditionAttendanceExporter.php:236, 259`; `EditionRegistrationExporter.php:948, 973`. This is exactly the `gotcha_stale_database_reads` bug class.

**M-6 — Theme hardcodes the active-status set in 5 places; the drift has grown from 3 sites since it was first tracked.** [F] `['announcement','open','full','in_progress']` literal at `page-klassikaal.php:42`, `page-online.php:68`, `partials/card-course.php:43`, `archive-sfwd-courses.php:38, :110`; zero theme usage of `OfferingStatus::activeValues()`, which was created on 2026-05-19 to kill this array.

**M-7 — No static analysis at all.** [F] No phpstan/psalm config; `composer lint` is Pint formatting only. [J] H-1 is precisely PHPStan-detectable territory, as are the known `validateSelections()` field-shape bugs.

**M-8 — ~90 MB of binary debris tracked in git.** [F] `bck/` (52 MB, 3,162 tracked files of a Kadence backup), five root SCORM/xAPI zips (~36 MB), six LTI screenshots, `themes/stride.zip`. Permanent clone weight.

**M-9 — Status labels duplicated outside Domain enums.** [F] `OfferingStatus::label()` / `RegistrationStatus::label()` exist, yet hand-rolled label maps live at `EditionAdminController.php:1354-1365`, `AdminAPIController.php:1920-1922`, `TrajectoryAdminController.php:637-642`.

**M-10 — Swallowed WP_Errors on admin money/duplication paths.** [F] `QuoteAdminController.php:540-543, 563-566` (discount silently no-ops), `EditionDuplicator.php:214-216` (duplicated edition silently loses sessions). One log line each.

**M-11 — LearnDash touched outside the adapter/helper at 7 sites (INV-6).** [F] Three have existing helper equivalents (`EnrollmentService.php:133-135`, `AuditBridge.php:192-193`, `PartnerAPIController.php:515-516`); four need new helper methods (`UserDashboardService.php:441-442`, `EditionAdminController.php:1201, 1212`, `EditionDetailsMetabox.php:228`, `EditionRegistrationMetabox.php:523`; plus raw SQL on `learndash_user_activity` at `PartnerAPIController.php:449-470`). **M-12 — Stale docs.** [F] README.md is stock Bedrock boilerplate with zero Stride content. 1. **Security posture is genuinely strong: zero Critical/High security findings.** Uniform `$wpdb->prepare`; the impersonation audit bypass is *fixed* (routes through `AuditService::record()` at `AdminAPIController.php:3246-3261, 3315-3329`, with CSPRNG tokens, admin-target blocking, and a caller-is-target check on `/end`); textbook Partner-API tenant isolation with explicit IDOR checks (`PartnerAPIController.php:350, 662-673`); consistent AJAX ownership checks; content-based MIME validation; CSV-injection neutralization on all export paths; no hardcoded secrets.

4. **Test quality where tests exist is high:** denial paths, exact error codes/HTTP statuses, DB-state assertions including negative ones (`PartnerAPIControllerTest.php:572-573`; `EnrollmentEdgeCest.php:116, 248, 295-311`).
5. **Concurrency is handled correctly where it matters:** `FOR UPDATE` row locks + re-checks on voucher redemption and enrollment capacity.
6. **Admin/ uses `BatchQueryHelper` consistently and well**; custom tables are properly indexed (verified live: composite indexes on all hot WHERE clauses).
7. **The codebase visibly absorbs its lessons** — prior audit fixes stuck, with explanatory comments at the fix sites.

## Improvement Strategy

### Theme 1 — Close the verification gap (explains CR-1, H-1, M-7, the invariant grep blind spot)
The repo has excellent verification *artifacts* and no verification *gate*: 1,800 tests, drift scanners, invariants — all run only when a human remembers. H-1 is the smoking gun: two dead-column queries survived because nothing executable ever touched those paths. **Target state:** CI fails when the unit suite fails; the integration suite runs in the existing devcontainer job; PHPStan level 1 (with a generated baseline) fails the build on new errors; INV-5's audit grep covers both theme prefixes. **Principle:** a check that doesn't run is documentation, not a check.

### Theme 2 — The read path must survive launch scale (CR-2, CR-3, H-3, H-4, H-8)
Writes are race-safe and batched; reads were built card-by-card. Every hot read surface (catalog, dashboard, notification badge, admin polls) does per-item queries against a stack with no object cache. **Target state:** card partials are pure renderers fed by one batched pre-pass; `getEnrollmentData`/`getQuoteData` memoized per request with nav flags from `EXISTS` queries; Redis object cache in production; notification unread-count cached and the audit table indexed for its read patterns. **Done means:** catalog page ≤ ~40 queries regardless of edition count; dashboard tab ≤ ~60; both asserted by a query-count smoke test so they can't regress.

### Theme 3 — One source of truth per business fact (H-5, H-6, M-5, M-6, M-9, M-11)
The recurring local bug class is "a constant/rule that has a canonical home gets re-typed at a call site" — VAT rate, status labels, active-status sets, table names, meta prefixes, LD calls. Each copy was individually reasonable; collectively they are where the next financial or catalog bug comes from. **Target state:** `QuoteCalculator` is the only place that knows 21%; `OfferingStatus::activeValues()` is the only active-set; `RegistrationRepository` is the only registrations-SQL author; stride-core owns its own date formatter. **Done means:** the invariant-check greps for each of these return only the canonical site, and those greps are added to `check-invariants.sh`.

### Theme 4 — Launch operability (H-2, M-1, M-2, M-4, M-12)
Small, sharp, launch-relevant correctness/ops items: a deploy procedure that exists, a CSV export that exports, an ENUM that accepts partner enrollments, PII access that leaves a trail, uploads that aren't world-readable, and a README that points newcomers at the real docs.

### Explicitly NOT fixing now (trade-offs)
- **AdminAPIController decomposition** — already planned post-launch (`project_unified_api_postlaunch`), previously declined, and the security audit found its authorization sound. Splitting a 3,510-line file weeks before launch is risk without payoff. We fix its two dead queries and add endpoint smoke tests instead.
- **Module dependency cycles (L-8)** — structural cleanup with no observed runtime harm; post-launch.
- **Partner API rate limiting (M-3)** — Partner API is post-launch scope per the standing audit; do it when partners onboard.
- **`ob_start+include` admin rendering, pass-throughs, scripts/ junk drawer** — cosmetic-to-minor; batch into M3 or with the eventual controller migration.
- **Playwright in CI** — needs a full WP environment with seeded data; defer until after PHPUnit/Codeception are wired (acceptance Cests give comparable coverage there).

### Definition of done (measurable)
1. CI fails on: unit-suite failure, integration-suite failure, PHPStan-baseline regression, Pint, invariants.
2. Zero verified-broken features (H-1 fixed with regression tests that would have caught them).
3. Catalog ≤ 40 queries / dashboard tab ≤ 60 queries, enforced by test.
4. Greps for `0.21`, the active-status literal, `vad_registrations` outside its repo, and `stride_format_` inside stride-core each return only canonical sites.
5. `make deploy-staging` works as documented, or site.yml documents what's real.
6. Repo clone shrinks by ~90 MB.
